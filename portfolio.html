About

A bit about me

I started my career at ElasticRun in 2021, straight into backend work on systems that were already running real operations. Since then I've spent five years building the platforms behind logistics, transportation, ERP, and day-to-day operations — the kind of systems where a slow query or a stuck queue shows up as orders not moving.

Most of my work sits in the backend: designing and shipping APIs, untangling distributed systems, tuning databases, and doing the unglamorous part — debugging things in production when they break. I like getting close to how a system actually behaves under load rather than how it's supposed to behave on paper, and a lot of my better wins have come from that.

Lately I've been focused on AI-assisted systems and operational intelligence — using retrieval and LLMs to make incident investigation and operations faster, while keeping the engineering grounded in real signals. I'm now looking for backend and backend-plus-AI roles where I can own platforms end to end.

Architecture Deep Dives

How the systems are built

Everything is collapsed by default — expand each section for why it exists, the data flow, and the engineering decisions behind it.

TMS exists to keep trips and shipments moving in real time — from a vehicle starting a trip to operations watching it on a live dashboard. The hard part isn't the workflow itself; it's keeping it fast and cheap as volume grows.

Data Flow

API Layer→ Services→ Kafka→ Workers→ Databases→ Dashboards

Major Components

  • API layer for trip, shipment, and vehicle operations
  • Kafka workers handling async, event-driven movement updates
  • MongoDB and PostgreSQL/TimescaleDB for live and historical data
  • Caching layer in front of hot read paths

Key Decisions & Scale

  • Moved heavy work off the request path into Kafka workers
  • Profiled and removed a CPU-bound bottleneck: 95% on 16 cores → 50% on 4 cores
  • Tuned queries and caching to bring API latency from 200ms to 50ms
  • Designed to absorb 15K+ transactions/day across 24K tracked trips

LMS is the backbone of order fulfilment — it carries an order from creation all the way to delivery. Because every step feeds the next, the system is built around events so services stay decoupled but consistent.

Data Flow

Orders→ Shipment Lifecycle→ Rider Assignment→ Location Tracking→ Notifications→ Operations Dashboard

Major Components

  • Modular backend split across ~10 focused services
  • Kafka coordinating shipment and assignment events
  • 256GB MySQL as the system of record
  • Redis caching for frequently read state

Key Decisions & Scale

  • Event-driven design to keep order state consistent across services
  • Database optimization on a large production MySQL without downtime
  • Sustains 30K+ orders/day and ~30K Kafka events/day

Both logistics and transportation need to know where riders and trips are — right now, and historically. Rather than each team solving this separately, location is a shared capability with the right storage for each access pattern.

Data Flow

Flutter App→ Tracking Service→ MongoDB / TimescaleDB→ Assignment Engine→ Dashboard

Major Components

  • Flutter rider app sending location updates
  • Go tracking service ingesting positions at scale
  • MongoDB for live geospatial queries; TimescaleDB/PostGIS for history

Key Decisions & Scale

  • Split live vs historical storage to match very different query patterns
  • Geospatial queries for proximity, routing, and assignment
  • Historical tracks support trip reconstruction and audits

When something breaks in production, most of the time goes into gathering context — logs, errors, metrics, and what's failed before. This assistant pulls that together and points to likely causes, so investigation starts from evidence instead of a blank page.

Data Flow

Logs / Errors / Metrics→ Normalization→ Chunking→ Embeddings→ Qdrant→ Retriever→ LLM→ Suggested Checks

Major Components

  • Ingestion and normalization of operational signals
  • Embeddings stored in Qdrant for semantic retrieval
  • Retriever + LLM that surface similar past incidents

Key Decisions & Scale

  • Evidence-first: answers stay grounded in real signals, not guesses
  • Retrieval of similar incidents to reuse past fixes
  • Cuts repeated manual debugging cycles for the team
